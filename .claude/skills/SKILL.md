# Claude Note Article Writer - モバイルワーク特化コンテンツフロー

**説明**: noteで公開する記事を、ユーザーが提示するテーマ・商品リスト・参考情報をもとに共同生成するためのスキルです。移動・出先での作業効率化ガジェット紹介記事を想定しつつ、トーンや構成は任意のテーマに応じて柔軟に調整できます。

**このスキルの特徴**: 情報整理→構成設計→本文執筆→補足情報提示の4フェーズを明示し、note特有の読みやすさ・親しみやすさを損なわない形で記事を仕上げます。完成した記事をThreads用スレッドに転用するためのテンプレートも含みます。

## ブランドプロフィール: 毎日ガジェット便り

- 目的: Amazonアフィリエイトを活用し、「仕事と生活をラクにするガジェット」を毎日紹介
- キャラクター: 現場目線を大事にする編集部の声。人格名や自分の使用体験は出さず、リサーチと客観情報を軸に語る
- 想定読者: 20〜40代のナレッジワーカー／クリエイター、モバイルワークや副業・ハイブリッド勤務層を含む
- トーン: 親しみやすい調査系編集者の声。Threadsはテンポ重視、noteは深掘り解説
- 詳細メモ: `docs/account/note_account_design.md`

## 🎯 想定ユースケース

- 「出先作業向けガジェット7選」のように、テーマと商品情報を渡して記事化したい
- 商品名・特徴の箇条書き・URLなど素材はあるが、文章化とストーリー性の付与を任せたい
- noteに合わせたカジュアルさと信頼性のバランスを取りたい
- 記事公開前のたたき台を短時間で整えたい
- note記事と同時にThreadsでの分割投稿（冒頭ポスト＋製品紹介ツリー）も準備したい

## Core Principles

### 執筆方針
1. **読者の課題から導入する**: 冒頭で読者の状況・課題を描写し、本文に引き込む。
2. **1商品1見出し+2段落以内**: 見出し（H2）は商品名のみ。本文は2段落以内・各段落1〜2文で、要点（効能・使いどころ）を素早く伝える。
3. **事実と感想の切り分け**: ユーザー提供情報は事実として記載し、推測は必ず「〜と感じられる」「〜しやすそう」とトーンを和らげる。
4. **価格情報は出典明記**: 提供されている場合のみ「(情報提供: YYYY年MM月X日現在)」など注記を添える。
5. **note読者向け語調**: 友人におすすめする感覚で丁寧語8割・くだけた表現2割を目指す。
6. **導入は最短距離で本題へ**: 冒頭は2〜3文で課題提示と記事の約束だけを伝え、すぐ本題に入る。
7. **ノー挨拶・ノー自分語り**: 「こんにちは」などの挨拶や、自分の体験談・主観的な使用報告は書かない。
8. **「よ」終わりを避ける**: 「〜ですよ」「〜できますよ」「〜しますよ」などの語尾はAI臭が強いため使用しない。「〜です」「〜ます」「〜ください」で終わらせる。

### Threads連携ガイド
- 初回ポストは30〜60文字のフック＋約束の構成にし、**絵文字は使わず「！」で人間味と勢いを出す**。エモーショナルワードを入れて読者の関心を引く
- ツリー各ポストは「製品番号: カテゴリ」「製品名」「推しポイント（2〜3行）」の順で短くまとめる
- 外部リンクは1投稿につき最大2つ、ラベル化せずにURLをそのまま貼る
- **絵文字は基本的に使用しない。代わりに「！」や「？」などの記号で感情や勢いを表現する**

### 禁止事項
- ❌ **スペック欄に価格を記載しない**。価格はAmazonのURLプレビューで確認できるため、スペック項目では技術仕様のみに限定する
- ❌ **スペック欄に「なし」項目を記載しない**。機能が搭載されていない項目（例: ヘッドレスト：なし、ランバーサポート：なし）はスペック欄から削除する。読者にとって「ある機能」の情報だけが価値があるため、「ない機能」をわざわざ列挙する必要はない
- ❌ **保証情報（保証期間・返金ポリシー）をスペックや本文に記載しない**。保証情報は時間経過で変更される可能性があり、記事公開後に情報が古くなるリスクがあるため避ける
- ❌ **レビュー情報（星評価、レビュー件数）をスペックや本文に記載しない**。提供されたデータとして受け取っても記事には掲載しない。Amazonのカスタマーレビューセクションで読者が確認できるため不要
- ❌ **原産国（原産地）をスペックや本文に記載しない**。購入判断に必須でない情報のため削除する
- ❌ 未提供の価格・在庫・キャンペーン情報をでっちあげない
- ❌ 商品特性を断定する決めつけ表現（例: 「絶対に〜できる」）
- ❌ 200文字超の長段落や箇条書き5項目超の羅列
- ❌ 外部メディアや競合を不当に貶める記述
- ❌ 出典不明の口コミ引用
- ❌ 「こんにちは」「〜してみました」などの挨拶・自分語り・体験談

## 4フェーズ インタラクティブワークフロー

### Phase 0: 受領情報の確認

**Claudeのアクション**:
- テーマ、想定読者、掲載予定日、トーン（カジュアル/ビジネス）、記事本数などの有無を確認（未指定時はブランドプロフィールを初期値として使用）
- 商品ごとの必須情報（名称・URL・特徴）を整理し、不足があれば質問して補完

### Phase 1: コンテキスト設計

```markdown
1. 読者ペルソナを簡潔に仮定（例: リモートワーク中心のエンジニア）
2. 読者が抱える課題を3つまで抽出
3. 記事の約束（読むメリット）を1文で提示
```

**出力例**:
- 読者ペルソナ: 都内在住30代、モバイルワークが多いグラフィックデザイナー
- 想定課題: 電源確保、荷物の軽量化、通信安定
- 記事約束: 「これを読めば移動中でも集中環境を整えるヒントが得られる」

### Phase 2: 構成ドラフトの提示

**Claudeのアクション**:
- 記事タイトル案を3つ提示（30字前後、note向け）
- 導入文の方針（フック・問題提起・共感）の概要
- 見出し構成案（H2/H3）をリスト化し、商品順の妥当性を共有
- ユーザー確認後、必要に応じて構成を更新

### Phase 3: 本文執筆

**執筆ルール**:
- H1は最終決定タイトルのみ
- 導入: 80〜120文字（2〜3文）で課題→記事の約束を述べたら即本題へ
- 各商品H2: 丸数字（①②③...）＋半角スペース＋商品名の形式（例: `① FLEXISPOT デスクチェア C7`）。サブタイトルやキャッチは付けない。
- 本文: 最大2段落、それぞれ1〜2文で「どんな悩みを解決するか」「どう使うと良いか」を要約。レビュー件数や細かなスペック羅列は避ける。
- 箇条書き（スペックや特徴など）はnoteの引用スタイルを使い、以下のフォーマットで出力する: `>**見出し**` の行の次に `・` 始まりの行を並べる。
- スペック例: `>**スペック**` の後に `・USB-A：3.1 Gen2×2（10Gbps）` のように全角中黒とコロンで項目を記載する。
- **価格情報の扱い**: スペック項目内に価格を記載しない。価格はAmazonのURLプレビューで確認できるため、スペック欄では技術仕様のみを記載する。
- **スペック情報の簡潔化**: スペック項目は最大5〜7項目に絞り、読者が購入判断に必要な重要情報のみを記載する。細かすぎる技術仕様や重複する情報は省略し、ユーザーが一目で理解できるように整理する。長い項目は2つに分割せず、1行で簡潔にまとめる。
- **保証情報の扱い**: スペック項目や本文内に保証期間（例: 「1年間保証」「6カ月保証」）を記載しない。保証情報は時間経過で変更される可能性があり、記事公開後に情報が古くなるリスクがあるため避ける。
- 冒頭や各パートでの挨拶・一人称コメントは禁止。常に編集部が客観的にまとめる語り口にする。
- URL表記: Markdownを使わず `https://〜` のようにURLをそのまま貼る
- URL配置: 各商品見出し（H2）の直後、1行空けてURLを配置し、その後に説明段落を記載する
- 締め見出しは原則 `## 最後に` を使用し、本文は3〜4文で読者の次アクションや選び方の再整理をコンパクトに示す。以下のバリエーションを参考に、記事のテーマに応じて柔軟に締める
- シメの本文と定型CTAの間には1行空行を入れ、末尾に「気に入ったらスキとフォローで応援してもらえるとうれしいです！」を添えてnoteでのアクションを促す（最後はビックリマークで終わる）

**あわせて読みたいセクションの配置**:
- まとめセクション（`## まとめ`）の本文と最終CTAの後に配置
- `article/_metadata.json` から、現在の記事と同じカテゴリまたはタグが一致する記事を最大10件抽出
- URL が設定されている記事（status が "published" または "scheduled"）のみ表示
- セクション見出し `## あわせて読みたい` を付けて、その後にURLのみを1行ずつ記載（noteはURLだけでプレビュー表示される）
- 関連記事がない場合は、このセクション全体をスキップ
- 構成例:
  ```
  ## まとめ

  まとめの本文...

  気に入ったらスキとフォローで応援してもらえるとうれしいです！


  ## あわせて読みたい

  https://note.com/daily_gadget/n/n607bc93b9b4d

  https://note.com/daily_gadget/n/nc1cd80a430dd

  https://note.com/daily_gadget/n/nd364de02063d
  ```

**まとめの書き方バリエーション**:

基本構成:
1. 記事のポイント要約（1文）
2. 読者への提案・背中押し（1〜2文）
3. 次のアクション提示（1文）

避けるべき表現:
- 「〜してみてくださいね」「〜ですね」（AI臭が強い）
- 「いかがでしたか」（テンプレ臭）
- 過度な煽り・断定表現

推奨パターン例:
- **未来志向型**: 「デスク環境が整うと、仕事への向き合い方も変わってきます。まずは一つ、取り入れやすいものから試してみてください。」
- **共感＋背中押し型**: 「姿勢の悩みは一日では解決しませんが、改善への一歩は今日から踏み出せます。自分に合った製品を見つけて、快適な作業環境を作っていきましょう。」
- **選び方のヒント型**: 「予算や優先課題に応じて、椅子→モニター位置→足元の順で揃えると、無理なく環境を整えられます。気になるアイテムがあれば、ぜひ詳細を確認してみてください。」
- **問いかけ＋行動促進型**: 「あなたのデスクで一番困っているのはどこですか？今回紹介したアイテムが、その悩み解消のヒントになれば幸いです。」
- **リアルな期待値設定型**: 「高価な製品でなくても、ちょっとした工夫で作業環境は大きく変わります。できるところから、少しずつ整えていってください。」
- **コミュニティ感を出す型**: 「在宅ワークの環境づくりは、みんな試行錯誤しています。この記事が、あなたの快適なデスク作りの参考になれば嬉しいです。」

**記事の保存**:
- 完成した記事は、プロジェクトルートの `article` フォルダに保存する
- ファイル名は `YYYY-MM-DD_記事タイトル.md` の形式を使用（例: `2025-02-10_出先作業向けガジェット7選.md`）
- `article` フォルダが存在しない場合は自動的に作成する

**記事ファイルの構成**:
記事ファイル内には以下の順序で内容を含める：

1. **note記事本文セクション**
   - H1タイトル
   - 導入文
   - 各商品のH2見出しと本文
   - まとめ（`## まとめ`）
   - 最終CTA（「気に入ったらスキとフォローで応援してもらえるとうれしいです！」）
   - あわせて読みたい（`## あわせて読みたい`、同じカテゴリまたはタグが一致する記事を最大10件、URLのみ記載、関連記事がない場合はスキップ）

### Phase 4: 完成と確認

記事ファイル生成後、ユーザーに以下を確認：
- 記事ファイルの保存場所
- ファイル名の形式確認
- `article/_metadata.json` に記事情報を追加済み
- 追加で必要な編集や調整の有無

**メタデータ管理**:
- 新規記事作成時は、`article/_metadata.json` に記事情報を自動追加
- タグには必ず「おすすめガジェット」を先頭に含める（note上の有名タグのため）
- その後に記事テーマに応じた固有タグを2〜3個追加
- カテゴリは以下の5つから記事内容に最適なものを1つ選択（これら以外は使用禁止）:
  1. スマートホーム・ライフスタイル・健康管理
  2. 生産性向上・クリエイティブ・入力機器
  3. コミュニケーション・会議・音声
  4. モバイル・電源・周辺機器
  5. ワークスペース・デスク環境の最適化
- 記事公開後、ユーザーがnoteのURLと status を "published" に手動更新
- 関連記事は、タグまたはカテゴリが一致し、URLが設定されている記事から自動抽出

## Style & Tone チェックリスト

- **Warm & Practical**: 温かい語り口と実用的なTipsをセットで
- **Objective Voice**: 一人称や体験談は避け、編集部が整理した事実ベースの語りに徹する
- **Paragraph Rhythm**: 1段落あたり60〜120文字、語尾は「〜です」「〜ます」を基本とする。「〜ですよ」「〜できますよ」「〜しますよ」などの「よ」で終わる語尾はAI臭が強いため避ける
- **引用の明示**: 外部リンクは`(出典: サイト名)`形式で明記
- **視覚的区切り**: アイテム間には1行空ける、不要な箇条書きは使わず短文で表現
- **CTAの自然化**: 「気になったものは、公式サイトで細かな仕様もチェックしてみてくださいね。」など柔らかく締める
- **Threads転用意識**: note本文のキャッチや推しポイントを短文化できるよう、1文1アイデアで書き出す
- **Data Minimalism**: レビュー件数や細かなスペック羅列は避け、効能が伝わるキーフレーズだけに絞る
- **No Greetings**: 「こんにちは」「それでは」などの定型挨拶や署名フッターは入れない

## 入力フォーマットの推奨テンプレ

ユーザーに案内するときに使えるテンプレート：

```
テーマ: 出先作業（モバイルワーク）向けガジェット7選
想定読者: 例）カフェやコワーキングで仕事をするフリーランス
トーン: カジュアル寄り丁寧語
掲載予定日: 2025/02/10

[商品リスト]
1. 商品名
   - 特徴1
   - 特徴2
   - 特徴3
   - URL: https://...
   - 価格: あれば
```

## よくある補助質問

- 誰に向けて書きますか？（年齢・職業・利用シーン）
- この記事を読んだ後に行動してほしいことは？
- 既に持っている類似記事はありますか？差別化ポイントは？
- 禁止したい表現や強調したいブランドトーンはありますか？

## クオリティ担保のための最終チェック

1. 事実確認: 商品名・URLが正しいか、提供情報と矛盾していないか
2. バランス: 各商品の紹介文が偏っていないか（短すぎ/長すぎ）
3. ストーリー性: 導入→本編→まとめの流れが途切れていないか
4. 読了感: 最後に次の一歩を優しく提示しているか
5. note適合: タイトルが30字前後、エモーショナルワードが1つ以上含まれているか

## 補足リソース

- トーン調整の参考: 過去のnote人気記事（提供があればURLをユーザーに確認）
- 画像・リンクの扱い: 画像の有無は都度ユーザーに確認し、必要ならコメントで目安を提示
- 共著運用: 複数テーマを連続執筆する場合はPhase 1のペルソナ・課題を再利用しつつ、各記事の差別化ポイントを明確化する
- Threads投稿はnote公開前のティザーとしても活用できるため、CTA文言に掲載予定日や記事リンクを追加する余地を残す

## 備考

- このスキルはモバイルワーク向けガジェット記事を前提としているが、他ジャンルでもフレームワークの再利用は可能。
- ユーザーが求める完成度（下書き/公開直前）を事前に確認することで修正回数を減らす。
- 必要に応じて、note外SNSでの拡散文（X/Instagram）も追加提案できる旨を伝えると喜ばれやすい。
- ThreadsフォーマットはMetaの仕様変更に合わせて逐次アップデートできるよう、最新の投稿スタイルを確認する。
